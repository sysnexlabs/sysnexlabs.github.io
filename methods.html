Hierarchical Development Flow

Our methodology follows a proven 8-level hierarchical decomposition that ensures complete traceability from stakeholder needs to implementation:

Level 1: Stakeholder Requirements (SYS.1)

Elicit stakeholder requirements, define use cases, formalize interfaces, and establish quality attributes. Foundation for all downstream development.

↓

Level 2: System Requirements (SYS.2)

Refine stakeholder requirements into system requirements using ISQ types, formal constraints, and requirement categorization.

↓

Level 3: Functional Architecture (SYS.3)

Design functional black box, decompose functions (Input, Process, Output, Safety/Security), and define functional effect chains.

↓

Level 4: Logical Architecture (SYS.3)

Define logical systems, decompose into subsystems, establish context systems, and create conversion interfaces.

↓

Level 5: Physical Architecture (SYS.3)

Aggregate software and hardware capabilities, define ECUs, physical elements, and establish physical system boundaries.

↓

Level 6: Hardware Level (HW.1-HW.4, HSI.1-HSI.4)

Decompose ECUs into hardware components (processors, memory, interfaces), define HSI components, and establish hardware integration.

↓

Level 7: Software Level (SWE.1-SWE.5)

Design software architecture, define capabilities and units, establish software interfaces, and create software integration.

↓

Level 8: Mechanical Level (MECH.1-MECH.4)

Define mechanical assemblies, components, interfaces, and establish mechanical integration and verification.

Traceability Strategy

Complete bidirectional traceability ensures every requirement is satisfied, verified, and properly allocated:

Requirements Traceability

Refinement chain from stakeholder requirements through system, physical, hardware, software, and mechanical requirements using explicit refine relationships.

  • Stakeholder Requirements (SR-*) → System Requirements (SYS-*)
  • System Requirements → Physical Requirements (PHYS-*)
  • Physical Requirements → Hardware/Software/Mechanical Requirements
  • HSI Requirements trace to both Hardware and Software levels

Architecture Traceability

Explicit trace relationships connect architecture levels, ensuring design decisions are properly documented and validated:

  • Physical Architecture → Logical Architecture → Functional Architecture
  • Hardware/Software/Mechanical Levels → Physical Architecture
  • HSI traces to both Hardware and Software levels

Requirements Satisfaction

Architecture elements satisfy requirements through explicit satisfy relationships, providing clear evidence of requirement fulfillment at all levels.

Verification & Validation

Comprehensive verification strategy ensures all requirements are validated and all architecture decisions are verified:

Verification Strategy

  • Requirements Verification: All requirements have verification cases with appropriate methods (Inspection, Analysis, Demonstration, Test)
  • Architecture Verification: Architecture-to-requirements and architecture-to-architecture traceability validation
  • Integration Verification: Integration test cases at all levels, including end-to-end integration
  • Unit Verification: Software unit test cases (SWE.4) and hardware component test cases

Verification Coverage

100% verification coverage across all requirement types: system, physical, hardware, software, mechanical, and HSI requirements. All verification cases linked to requirements via verify relationships.

Key Principles & Best Practices

Requirements Engineering

  • Use ISQ quantity types instead of generic Real for quantitative requirements
  • Use require blocks for formal constraints
  • Always define subject relationships
  • Use refine relationships for requirement decomposition

Architecture Design

  • Use interface inheritance for consistency across levels
  • Separate concerns into dedicated files and packages
  • Define ports and interfaces explicitly
  • Establish trace relationships between architecture levels

Integration

  • Use part usages for composition
  • Use bind and connect for connections
  • Define effect chains for data/control flow

Verification

  • Define verification cases for all requirements
  • Use appropriate verification methods (Test, Inspection, Analysis, Demonstration)
  • Link verification cases to requirements via verify relationships
